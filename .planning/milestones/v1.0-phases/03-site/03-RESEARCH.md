# Phase 3: Site - Research

**Researched:** 2026-03-06
**Domain:** Astro static site, Chart.js dual-line chart, Playwright OG screenshot, GitHub Pages
**Confidence:** HIGH

## Summary

Phase 3 builds the complete Estonian awareness site on top of the data pipeline completed in Phases 1-2. The core challenge is assembling three independent concerns — Astro-based page scaffolding, Chart.js visualization, and Playwright-based OG image generation — into a single `npm run build` that the existing `deploy.yml` can invoke.

The Astro framework is the right choice for a static site with no dynamic routing: it ships zero JS by default, bundles Chart.js into a hashed script automatically, and outputs to `dist/` which the existing workflow already points to. Chart.js 4 with `chart.js/auto` is appropriate here because bundle size is not a concern for a single-page awareness site. The OG image is generated by a post-build Node.js/TypeScript script that launches a local HTTP server over `dist/`, navigates Playwright to `localhost`, takes a 1200×630 screenshot, and writes `dist/og-preview.png` before the upload artifact step runs.

The main sequencing risk is that the OG screenshot script must run AFTER `astro build` but write INTO `dist/` BEFORE `actions/upload-pages-artifact@v4` is called. The deploy.yml already has a Phase 3 insert comment for exactly this slot.

**Primary recommendation:** Install Astro and Chart.js, write a single `src/pages/index.astro` with an inline `<script>` block, add a `scripts/og-screenshot.ts` post-build step, wire `deploy.yml`, and update `package.json` with a `build` script that chains both.

### Pattern 1: Astro page with inline Chart.js script
**What:** A single `.astro` file imports `chart-data.json` at build time via a top-level `import`, inlines the data as a JSON script tag, and initializes Chart.js in a `<script>` tag that Astro bundles.
**When to use:** Single-page static sites with no framework UI components; avoids React/Vue overhead.

```astro
---
// Source: https://docs.astro.build/en/guides/client-side-scripts/
import chartData from '../data/chart-data.json';
const { months } = chartData;

// Compute summary sentence at build time
const closedMonths = months.filter(m => m.scamEur !== null && m.scamEur > 0 && m.pensionEur > 0);
const lastClosed = closedMonths[closedMonths.length - 1];
const pct = lastClosed
  ? ((lastClosed.scamEur / lastClosed.pensionEur) * 100).toFixed(1)
  : null;
---

<script define:vars={{ months }}>
  import Chart from 'chart.js/auto';

  const labels = months.map(m => m.month);
  const scam = months.map(m => m.scamEur);
  const pension = months.map(m => m.pensionEur);

  new Chart(document.getElementById('chart'), {
    type: 'line',
    data: {
      labels,
      datasets: [
        {
          label: 'Pensionimaksed',
          data: pension,
          borderColor: '#2563eb',
          backgroundColor: 'transparent',
          spanGaps: true,
        },
        {
          label: 'Pettusekahju',
          data: scam,
          borderColor: '#dc2626',
          backgroundColor: 'transparent',
          spanGaps: true,
        }
      ]
    },
    options: {
      responsive: true,
      maintainAspectRatio: false,
      interaction: {
        mode: 'index',
        intersect: false,
      },
      plugins: {
        tooltip: {
          callbacks: {
            label: ctx => `${ctx.dataset.label}: ${ctx.raw?.toLocaleString('et-EE')} €`
          }
        }
      },
      scales: {
        y: {
          ticks: {
            callback: v => `${(v / 1e6).toFixed(0)} M€`
          }
        }
      }
    }
  });
</script>
```

**Critical note on `define:vars`:** When using `define:vars`, Astro forces the script to `is:inline` mode, which bypasses Astro's bundler. This means `import Chart from 'chart.js/auto'` inside the same `<script>` block will NOT work with `define:vars`. Instead, pass data via a `<script type="application/json" id="chart-data">` element and read it from the DOM, OR use two separate script blocks (one `is:inline` for data, one normal for Chart.js import).

**Correct pattern:**
```astro
---
import chartData from '../data/chart-data.json';
---

<!-- Data passed via DOM, not define:vars -->
<script type="application/json" id="chart-data" set:text={JSON.stringify(chartData)}></script>

<script>
  import Chart from 'chart.js/auto';
  const data = JSON.parse(document.getElementById('chart-data').textContent);
  // ... Chart initialization
</script>
```

### Pattern 2: Post-build OG screenshot via Playwright
**What:** A TypeScript script that (1) starts a Node `http.createServer` pointing at `dist/`, (2) navigates Playwright to `http://localhost:PORT`, (3) waits for the chart canvas to render, (4) takes a 1200×630 screenshot, (5) saves it to `dist/og-preview.png`, (6) kills the server.
**When to use:** Any static site needing a chart-rendered OG image without a CI image generation service.

```typescript
// Source: https://playwright.dev/docs/library
import { chromium } from 'playwright';
import * as http from 'http';
import * as fs from 'fs';
import * as path from 'path';

const DIST = path.resolve('dist');
const PORT = 4321;

// Minimal static file server for dist/
const server = http.createServer((req, res) => {
  const filePath = path.join(DIST, req.url === '/' ? 'index.html' : req.url!);
  try {
    const data = fs.readFileSync(filePath);
    res.end(data);
  } catch {
    res.writeHead(404).end();
  }
});

await new Promise<void>(resolve => server.listen(PORT, resolve));

const browser = await chromium.launch();
const context = await browser.newContext({
  viewport: { width: 1200, height: 630 }
});
const page = await context.newPage();
await page.goto(`http://localhost:${PORT}`);
await page.waitForSelector('canvas');   // wait for Chart.js to render
await page.screenshot({ path: path.join(DIST, 'og-preview.png') });

await browser.close();
server.close();
```

### Pattern 3: Astro GitHub Pages configuration
**What:** `astro.config.mjs` must set `site` and optionally `base` for correct absolute URLs on GitHub Pages.
**When to use:** Any GitHub Pages repo where the site is NOT at `<username>.github.io`.

```javascript

export default defineConfig({
  site: 'https://<username>.github.io',
  base: '/<repo-name>',  // only if NOT username.github.io repo

```

**Note:** The OG `og:image` URL must be absolute (`site + base + '/og-preview.png'`). Use `Astro.site` + `Astro.url` in the frontmatter to construct this.

### Pattern 4: `package.json` build script chaining
```json
{
  "scripts": {
    "dev": "astro dev",
    "build": "astro build && tsx scripts/og-screenshot.ts",
    "preview": "astro preview"
  }
}
```

The `&&` ensures `og-screenshot.ts` only runs if `astro build` succeeded.

### Anti-Patterns to Avoid
- **`define:vars` with ES module imports:** Astro's `define:vars` forces `is:inline` which breaks `import` statements. Pass data via a `<script type="application/json">` tag instead.
- **Dual y-axes:** User locked this out. A single shared EUR axis is intentional — the visual gap is the message.
- **`pensionEur: 0` months in the "last closed" calculation:** The current month (March 2026) has `pensionEur: 0` because pension data is not yet published. The summary sentence logic must skip months where `pensionEur === 0` to find the true last complete month.
- **Screenshotting before Chart.js renders:** `await page.goto(url)` alone is not sufficient — Chart.js renders asynchronously on canvas. Use `await page.waitForSelector('canvas')` and consider a short `waitForTimeout` or a sentinel element added after chart init.
- **Relative OG image URL:** Facebook and Telegram require an absolute URL in `og:image`. Using a relative path will result in the image not loading in social previews.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| EUR number formatting | Custom formatter | `Number.toLocaleString('et-EE')` | Handles thousands separators, decimal comma in Estonian locale |
| Static file serving for screenshot | Complex Express server | 20-line Node `http.createServer` | No dependency; `dist/` is only static files |
| Chart rendering | Custom SVG chart | Chart.js 4 | Accessibility, responsive canvas, touch events, tooltip system |
| OG image service | External API (cloudinary, vercel/og) | Playwright screenshot | Already installed; no external dependency or API key needed |
| Month name localization | Hardcoded string array | `new Date('2026-02-01').toLocaleDateString('et-EE', { month: 'long' })` for nominative, then adjust to genitive | Browser/Node Intl API handles nominative; genitive suffix is simple rule for Estonian months |

**Key insight:** Node's `Intl` API provides nominative month names in Estonian. For the genitive suffix in "Petturid varastasid X% [month-genitive] pensionimaksetest", the pattern is: append `'i'` to the nominative for most months, with a lookup table for irregulars (märts → märtsi, mai → mai, august → augusti, oktoober → oktoobri, november → novembri, detsember → detsembri).

## Common Pitfalls

### Pitfall 1: `define:vars` breaks ES module imports
**What goes wrong:** Using `define:vars={{ months }}` in a `<script>` block prevents Astro from bundling it as an ES module. Any `import` inside that block is silently ignored or throws at runtime.
**Why it happens:** Astro adds `is:inline` automatically when `define:vars` is present, which bypasses Vite bundling.
**How to avoid:** Put chart data in a `<script type="application/json" id="chart-data">` element and read it via `JSON.parse(document.getElementById('chart-data').textContent)` in a separate normal `<script>` block.
**Warning signs:** Chart.js fails to import; `Chart is not defined` error in browser console.

### Pitfall 2: OG screenshot captures blank/loading canvas
**What goes wrong:** Playwright screenshots the page before Chart.js finishes drawing.
**Why it happens:** `page.goto()` resolves at `load` event, but canvas rendering happens after `DOMContentLoaded` in a `requestAnimationFrame`.
**How to avoid:** After `page.goto()`, add `await page.waitForFunction(() => document.querySelector('canvas')?.getContext('2d') !== null)`. Alternatively, add a `data-ready` attribute to the canvas after chart init and wait for that.
**Warning signs:** OG image is blank white or shows only axes with no data.

### Pitfall 3: `pensionEur: 0` in most-recent-month calculation
**What goes wrong:** The current/running month in `chart-data.json` has `pensionEur: 0` because pension API data for the current month is not yet available. Using the last array entry for the summary sentence produces `0%` or `Infinity%`.
**Why it happens:** The scraper defaults missing pension months to `pensionEur: 0` (confirmed in project decisions).
**How to avoid:** Filter `months` to only entries where both `scamEur !== null && scamEur > 0 && pensionEur > 0` before selecting the last element.
**Warning signs:** Summary sentence says "0%" or is blank.

### Pitfall 4: Relative path for og:image
**What goes wrong:** Social platforms (Facebook, Telegram) don't resolve relative URLs in `og:image`.
**Why it happens:** `og:image` requires an absolute URL per the OpenGraph spec.
**How to avoid:** Construct absolute URL using `Astro.site` and the base path. Example: `${Astro.site}og-preview.png`.
**Warning signs:** Link preview on Facebook/Telegram shows no image.

### Pitfall 5: Missing Chromium browser on CI
**What goes wrong:** `npx playwright install` is not run in the deploy.yml build job; `chromium.launch()` throws "Executable doesn't exist".
**Why it happens:** Playwright installs the library but not the browser binaries unless explicitly requested.
**How to avoid:** Add `npx playwright install chromium --with-deps` to the build job after `npm ci` and before `npm run build`.
**Warning signs:** Build fails with `browserType.launch: Executable doesn't exist`.

### Pitfall 6: Astro `base` path missing from OG image URL on GitHub Pages
**What goes wrong:** If the repo is a project page (not `<username>.github.io`), Astro serves from `/<repo-name>/`. The OG image at `https://<username>.github.io/og-preview.png` returns 404.
**Why it happens:** Without `base` configured, internal asset URLs are root-relative and incorrect.
**How to avoid:** Set `base: '/<repo-name>'` in `astro.config.mjs` and reference the OG image as `${Astro.site}${Astro.url.pathname}og-preview.png` or use `import.meta.env.BASE_URL`.

### Playwright screenshot (TypeScript, post-build)
```typescript
// Source: https://playwright.dev/docs/library
import { chromium } from 'playwright';
import * as http from 'node:http';
import * as fs from 'node:fs';
import * as path from 'node:path';

const DIST = path.resolve('dist');
const PORT = 4173;

// Minimal static server for dist/
const server = http.createServer((req, res) => {
  const rel = req.url === '/' ? 'index.html' : req.url!.replace(/^\//, '');
  const file = path.join(DIST, rel);
  try {
    res.end(fs.readFileSync(file));
  } catch {
    res.writeHead(404).end('Not found');
  }
});

await new Promise<void>(r => server.listen(PORT, r));

const browser = await chromium.launch();
const context = await browser.newContext({
  viewport: { width: 1200, height: 630 }
});
const page = await context.newPage();
await page.goto(`http://localhost:${PORT}`);
// Wait for canvas to have actual drawn content
await page.waitForFunction(() => {
  const c = document.querySelector('canvas') as HTMLCanvasElement | null;
  if (!c) return false;
  const ctx = c.getContext('2d');
  if (!ctx) return false;
  const d = ctx.getImageData(0, 0, c.width, c.height).data;
  return d.some(v => v !== 0);
});
await page.screenshot({
  path: path.join(DIST, 'og-preview.png'),
  clip: { x: 0, y: 0, width: 1200, height: 630 }
});

await browser.close();
server.close();
```

### Estonian month genitive lookup for summary sentence
```typescript
// Genitive forms for "Petturid varastasid X% [month-genitive] pensionimaksetest"
const MONTH_GENITIVE: Record<string, string> = {
  '01': 'jaanuari',
  '02': 'veebruari',
  '03': 'märtsi',
  '04': 'aprilli',
  '05': 'mai',
  '06': 'juuni',
  '07': 'juuli',
  '08': 'augusti',
  '09': 'septembri',
  '10': 'oktoobri',
  '11': 'novembri',
  '12': 'detsembri',
};

function summarySentence(months: { month: string; scamEur: number | null; pensionEur: number }[]): string | null {
  const closed = months.filter(m => m.scamEur !== null && m.scamEur > 0 && m.pensionEur > 0);
  const last = closed[closed.length - 1];
  if (!last) return null;
  const [year, mon] = last.month.split('-');
  const pct = ((last.scamEur! / last.pensionEur) * 100).toFixed(1);
  const monthEt = MONTH_GENITIVE[mon] ?? mon;
  return `Petturid varastasid ${pct}% ${monthEt} ${year} pensionimaksetest`;
}
```

## Open Questions

1. **GitHub Pages repo URL pattern (`<username>.github.io` vs project page)**
   - What we know: `astro.config.mjs` `base` is required for project pages, not for user pages
   - What's unclear: The actual GitHub repo URL is not documented in the project files
   - Recommendation: Check repo name; if it matches `<username>.github.io`, omit `base`. Otherwise set `base: '/<repo-name>'`. The planner should add a task to verify and set these values.

2. **Playwright Chromium binary on CI — `--with-deps` flag availability**
   - What we know: `npx playwright install chromium --with-deps` installs system dependencies on Ubuntu
   - What's unclear: Whether the GitHub Actions ubuntu-latest runner has all required system libraries
   - Recommendation: Use `--with-deps` which handles `libnss3` and other Chromium deps on Ubuntu. This is the standard Playwright CI pattern.

3. **Canvas readiness detection for OG screenshot**
   - What we know: `page.waitForSelector('canvas')` is insufficient; Chart.js draws asynchronously
   - What's unclear: Whether a `waitForFunction` pixel check is fast enough vs adding a sentinel DOM attribute
   - Recommendation: Add `data-chart-ready="true"` to the canvas element in the Chart.js `options.animation.onComplete` callback; then `await page.waitForSelector('canvas[data-chart-ready]')` in the screenshot script. This is more reliable than pixel inspection.
